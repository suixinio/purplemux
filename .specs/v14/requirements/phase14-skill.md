# Phase 14 — 스킬 (Skill) PRD

## 목표

자주 쓰는 Claude Code 명령을 원클릭으로 실행하는 스킬 시스템을 제공하는 것.

현재(Phase 13까지)는 Claude Code에 매번 텍스트를 직접 입력해야 한다. 반복적으로 사용하는 명령(커밋, 코드 리뷰 등)을 스킬로 정의해두면, 입력창 위에 표시되는 suggest 버튼을 클릭하는 것만으로 실행할 수 있다.

## 완료 조건

입력창 위 suggest에서 "커밋하기"를 클릭하면 자동으로 변경사항을 분석하고 커밋 실행. 설정에서 스킬 on/off 가능.

---

## 현재 상태 (Phase 13 완료)

### 이미 구현된 것

- Web 입력창: 텍스트 입력 + Enter로 Claude Code에 전송
- CLI 상태 감지 (idle/busy/inactive)
- Claude Code Panel + 타임라인

### Phase 14에서 추가할 것

- 스킬 정의 포맷 (마크다운 + YAML front matter)
- 빌트인 스킬: "커밋하기"
- 입력창 위 suggest 버튼
- 스킬 실행 → 프롬프트 자동 입력 → 터미널 전송
- 설정 페이지에서 스킬 on/off

## 요구사항

### REQ-1: 스킬 정의 포맷

각 스킬은 마크다운 파일 하나로 정의한다.

- YAML front matter:
  - `name`: 스킬 이름 (UI 표시용)
  - `description`: 스킬 설명
  - `allowed-tools`: Claude Code에서 사용할 도구 목록 (선택적)
- 본문: 프롬프트 텍스트
  - `` !`command` `` 문법으로 동적 컨텍스트 주입 가능
  - 예: `` !`git status` `` → 실행 결과를 프롬프트에 삽입
- Claude Code 플러그인 포맷 참고

예시 — `commit.md`:

```markdown
---
name: 커밋하기
description: 변경사항을 분석하고 커밋합니다
allowed-tools:
  - Bash
  - Read
---

다음 변경사항을 분석하고 적절한 커밋 메시지를 작성하여 커밋해주세요.

!`git status`
!`git diff HEAD`
!`git log --oneline -10`
```

### REQ-2: 빌트인 스킬

1차 구현에서는 "커밋하기" 스킬 하나만 빌트인으로 탑재한다.

- `git status`, `git diff HEAD`, `git log --oneline -10` 결과를 자동 수집
- 변경사항 분석 후 스테이징 + 커밋 메시지 생성 + 커밋을 한 번에 실행
- 스킬 파일은 서버에 내장 (별도 설치 불필요)

### REQ-3: 스킬 실행 흐름

스킬 실행 시 프롬프트를 입력창에 자동 입력하고 터미널로 전송한다.

- suggest 버튼 클릭 → 스킬 프롬프트 생성
  - `` !`command` `` 부분은 서버에서 실행하여 결과로 치환
- 생성된 프롬프트를 입력창에 표시 (사용자가 확인/수정 가능)
  - 또는 즉시 전송 (확인 없이)
- 전송: 기존 Web 입력창의 전송 로직 재활용 (MSG_STDIN → PTY write)

### REQ-4: 입력창 위 Suggest UI

입력창 상단에 스킬 버튼을 suggest 형태로 노출한다.

- 위치: 입력창 바로 위, 가로로 나열
- 각 버튼: 스킬 이름 텍스트 (예: "커밋하기")
- 클릭 → 해당 스킬 즉시 실행
- Claude Code가 idle 상태일 때만 표시
- busy/inactive 상태에서는 숨김
- 스킬이 비활성화(off)되면 suggest에서 숨김

### REQ-5: 설정 페이지 — 스킬 관리

설정 페이지에서 스킬 목록을 표시하고 켜기/끄기를 관리한다.

- 스킬 목록: 이름 + 설명 + on/off 토글
- 켜진 스킬만 입력창 위 suggest에 표시
- 설정 저장: `~/.purple-terminal/` 하위 JSON 파일
- 기본값: 모든 빌트인 스킬 켜짐

---

## 비기능 요구사항

### NFR-1: 스킬 실행 속도

`` !`command` `` 치환은 서버에서 실행 — 명령어 실행 시간에 의존하지만, `git status` 등은 수 ms.

### NFR-2: 기존 기능 영향 없음

스킬 시스템 추가가 기존 입력창, 터미널, 타임라인 기능에 영향을 주지 않아야 한다.

### NFR-3: 확장성

1차는 빌트인 스킬 1개이지만, 추후 사용자 커스텀 스킬을 쉽게 추가할 수 있는 구조로 설계한다.

---

## 범위 제외 (Phase 14에서 하지 않는 것)

| 항목 | 사유 |
|---|---|
| 사용자 커스텀 스킬 추가 (UI 편집기) | 추후 확장 |
| Workspace별 스킬 세트 구성 | 추후 확장 |
| 스킬 마켓 / 공유 | 범위 과대 |
| 스킬 파라미터 입력 (실행 시 변수 주입) | 추후 확장 |
| 스킬 실행 기록/히스토리 | 추후 확장 |

## 기술 구성

### 스킬 실행 흐름

```
Suggest 버튼 클릭
├── 클라이언트 → 서버: 스킬 실행 요청 (스킬 ID)
├── 서버: 스킬 마크다운 파일 읽기
├── 서버: !`command` 부분을 실행하여 결과로 치환
├── 서버 → 클라이언트: 완성된 프롬프트 텍스트
├── 클라이언트: 입력창에 프롬프트 표시 (또는 즉시 전송)
└── 클라이언트: 기존 입력창 전송 로직으로 PTY에 write
```

### 설정 저장

```
~/.purple-terminal/skills.json
{
  "skills": {
    "commit": { "enabled": true }
  }
}
```

## 검증 시나리오

1. **Suggest 표시**: Claude Code idle 상태에서 입력창 위에 "커밋하기" 버튼 표시
2. **스킬 실행**: "커밋하기" 클릭 → git 상태 수집 → 프롬프트 생성 → Claude Code에 전송
3. **커밋 완료**: Claude Code가 변경사항 분석 후 커밋 실행
4. **busy 상태**: Claude Code 처리 중에는 suggest 숨김
5. **설정 off**: 설정에서 "커밋하기" off → suggest에서 사라짐
6. **설정 on**: 다시 on → suggest에 복귀
7. **inactive 상태**: Claude Code 비활성 시 suggest 숨김
8. **모바일**: 모바일에서도 입력창 위 suggest 동일하게 표시
9. **기존 기능**: 스킬 추가 후에도 입력창, 터미널, 타임라인 정상 동작
10. **에러 처리**: git이 없는 디렉토리에서 커밋 스킬 실행 → 적절한 에러 메시지
