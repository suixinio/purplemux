## 페이지 목록 (도출)

v13은 기존 데스크톱 UI를 유지하면서, 모바일 뷰포트에 맞는 별도 레이아웃과 네비게이션을 추가한다.

| 페이지 | 설명 | 우선순위 |
|---|---|---|
| 모바일 레이아웃 | 단일 Surface 전체 화면 + 상단 네비게이션 바 | P0 |
| 모바일 트리 탐색 | 햄버거 메뉴 → Workspace/Pane/Surface 트리 네비게이션 | P0 |
| 모바일 Claude Code | 타임라인/터미널 탭 전환 + 하단 고정 입력창 | P0 |
| 모바일 통계 | `/stats` 페이지의 모바일 반응형 레이아웃 | P1 |

## 주요 요구사항

### 반응형 분기

- 뷰포트 768px 기준으로 데스크톱/모바일 레이아웃 자동 전환
- 동일 URL, 동일 서버 — 반응형 웹으로 구현 (별도 앱 없음)
- JavaScript 기반 감지 (`useMediaQuery` 또는 `window.matchMedia`)
  - CSS 미디어 쿼리만으로는 컴포넌트 트리 자체가 달라지므로 JS 분기 필요
- 화면 회전(가로/세로) 시 자동 재조정 (resize 이벤트 debounce)
- 데스크톱 레이아웃은 **일절 변경 없음** — 모바일은 별도 컴포넌트 트리

### 모바일 화면 구조

- Pane 분할 없이 **한 번에 하나의 Surface만 전체 화면**으로 표시
- 데스크톱의 Pane 레이아웃 데이터는 그대로 유지하되, 모바일에서는 무시하고 단일 Surface 렌더링
- 상단: 네비게이션 바 (햄버거 메뉴 + 현재 위치 breadcrumb)
- 하단: 탭 인디케이터 도트 (같은 Pane 내 Surface 위치)
- Surface 내용: 터미널 또는 Claude Code Panel — 데스크톱과 동일한 기능

### 트리 탐색 네비게이션

- 햄버거 메뉴 → 전체 화면 오버레이 또는 슬라이드인 시트
- Workspace 목록 (데스크톱 사이드바 대체)
- Workspace 선택 → 하위 Pane/Surface 트리 펼침 (아코디언)
- Surface 선택 → 메뉴 닫힘 + 해당 터미널 전체 화면 전환
- 현재 선택된 Surface 하이라이트 (`●` 마커)
- 하단: 통계 페이지 링크

### 모바일 터미널 최적화

- xterm.js cols/rows를 모바일 뷰포트에 맞게 재계산
  - 세로 모드: ~40 cols, 가로 모드: ~80 cols
  - tmux 세션에 resize 전달 (기존 resize 로직 재활용)
- 모바일 키보드 표시 시 터미널 영역 자동 리사이즈
  - `visualViewport` API 또는 resize 이벤트로 키보드 높이 감지
  - 터미널 높이 = 뷰포트 높이 - 네비게이션 바 - 키보드 높이
- 터미널 폰트: 모바일에서 가독성 확보 (최소 12px)

### 모바일 Claude Code Panel

데스크톱의 타임라인+터미널 분할 대신, **탭 형태로 전환**한다:

| 탭 | 내용 |
|---|---|
| 타임라인 (기본) | 타임라인 뷰 전체 화면 |
| 터미널 | xterm.js 전체 화면 |

- 상단 네비게이션 바 우측에 터미널 토글 버튼 (`⌨` 아이콘)
- 입력창: 화면 하단 고정 (모바일 키보드 위)
- 세션 메타 바: 컴팩트 모드만 (상세는 별도 바텀 시트)
- 세션 목록: 전체 화면 리스트 (트리 탐색과 별도)

### 모바일 생성 기능

모바일에서도 최소한의 생성/관리 기능을 제공한다.

- **새 탭 생성**: 트리 탐색 메뉴에서 Pane 하위에 "+" 버튼 → 새 Surface(탭) 생성
- **탭 닫기**: 현재 Surface를 닫는 버튼 (네비게이션 바 또는 트리 메뉴)
- Workspace/Pane 생성은 데스크톱에서만 (모바일에서는 조회/전환만)

### 통계 페이지 모바일 대응

- `/stats` 페이지: 단일 컬럼 레이아웃 (너비 100%)
- 차트: `ResponsiveContainer`(recharts)로 모바일 너비에 반응
- 기간 필터: 가로 스크롤 가능한 탭 버튼 그룹
- 섹션 간 간격 축소, 카드 패딩 줄임

## 제약 조건 / 참고 사항

### 기술적 제약

- **컴포넌트 분기**: 모바일/데스크톱은 별도 컴포넌트 트리. 공통 로직(훅, 상태)은 공유하되 렌더링은 분리
- **WebSocket 공유**: 모바일/데스크톱 모두 동일한 터미널/타임라인 WebSocket 사용. 모바일에서는 현재 보이는 Surface만 활성, 나머지는 백그라운드 유지
- **모바일 키보드**: 키보드 높이 감지는 최소한의 노력으로 처리. 키보드가 올라오면 터미널이 위로 밀려나는 것은 허용. 완벽한 키보드 대응보다 안정성 우선

### UX 고려사항

- **첫 진입**: 모바일 접속 시 마지막으로 활성이던 Workspace의 첫 번째 Surface를 자동 선택
- **전환**: Surface 전환은 명시적 클릭으로만 (스와이프 없음)
- **터치 타겟**: 모든 터치 타겟은 최소 44x44px (Apple HIG)
- **Safe Area**: 노치/Dynamic Island/홈바 영역 대응 (`env(safe-area-inset-*)`)
- **스크롤 바운스**: iOS Safari의 오버스크롤 바운스를 적절히 제어 (터미널 영역에서는 비활성)

### 성능

- 모바일에서도 터미널 입출력 반응성 동일 (WebSocket 기반이므로 네트워크 의존)
- Surface 전환 시 xterm.js 인스턴스를 미리 마운트하여 즉시 표시
- 메모리: 모바일에서 동시에 마운트하는 xterm.js 인스턴스 수 제한 (현재 Pane의 Surface만)

## 확정 사항 (미확인 → 해결)

- [x] 스와이프 → 사용하지 않음. 모바일에서는 명시적 클릭으로 Surface 전환. 잘못된 스와이프로 화면이 넘어가는 문제 방지
- [x] 모바일 키보드 → 최소한의 노력으로 처리. 키보드가 올라오면 터미널이 밀려나는 것은 허용
