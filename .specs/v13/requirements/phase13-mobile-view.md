# Phase 13 — 모바일 뷰 PRD

## 목표

모바일 브라우저에서 터미널을 단순하게 조회하고 조작할 수 있게 하는 것.

현재(Phase 12까지)는 데스크톱 브라우저에 최적화되어 있어, 모바일에서는 Pane 분할, 사이드바, 탭 바 등이 좁은 화면에 맞지 않는다. Phase 13에서는 모바일 뷰포트에 맞춘 전용 레이아웃을 제공하여, 이동 중에도 터미널과 Claude Code를 사용할 수 있게 한다.

## 완료 조건

모바일 브라우저에서 Workspace/Pane/Surface를 트리로 탐색하고, 선택한 터미널을 전체 화면으로 사용 가능.

---

## 현재 상태 (Phase 12 완료)

### 이미 구현된 것

- 데스크톱 레이아웃: 사이드바 + Pane 분할 + 탭 바 + 터미널/Claude Code Panel
- 모든 기능이 데스크톱 뷰포트 기준으로 설계됨
- 반응형 대응 없음 (모바일에서 사용 불편)

### Phase 13에서 추가할 것

- 모바일 전용 레이아웃 (Pane 분할 없이 단일 Surface 전체 화면)
- Workspace → Pane → Surface 트리 탐색 네비게이션
- 모바일 최적화 (폰트, 키보드, 스와이프)
- 뷰포트 기반 자동 분기 (데스크톱 ↔ 모바일)

---

## 요구사항

### REQ-1: 반응형 분기

뷰포트 크기에 따라 데스크톱/모바일 레이아웃을 자동 전환한다.

- 동일 URL, 동일 서버 — 별도 앱 없이 반응형 웹으로 구현
- 분기 기준: 뷰포트 너비 768px 미만 → 모바일 레이아웃
- 768px 이상 → 기존 데스크톱 레이아웃 (변경 없음)
- CSS 미디어 쿼리 또는 JavaScript 기반 감지
- 화면 회전(가로/세로) 시 자동 재조정

### REQ-2: 모바일 화면 구조

모바일에서는 Pane 분할 없이 한 번에 하나의 Surface만 전체 화면으로 표시한다.

- 데스크톱의 Pane 분할을 무시하고, 단일 Surface를 전체 화면으로 렌더링
- Surface 내용은 데스크톱과 동일 (터미널 또는 Claude Code Panel)
- 상단에 네비게이션 바 (현재 위치 + 메뉴 버튼)
- 하단 고정: 입력창 (Claude Code 모드일 때)

### REQ-3: 트리 탐색 네비게이션

Workspace → Pane → Surface를 트리 구조로 탐색하여 선택한다.

```
Workspace (메뉴)
├── Pane 1
│   ├── Surface A
│   └── Surface B
└── Pane 2
    └── Surface C
```

- 햄버거 메뉴 → Workspace 목록 (데스크톱의 사이드바 대체)
- Workspace 선택 시 하위 Pane/Surface 트리 펼침
- Surface 선택 시 해당 터미널 전체 화면 전환
- 현재 보고 있는 Surface의 위치를 네비게이션 바에 표시 (breadcrumb)

### REQ-4: 모바일 터미널 최적화

모바일 뷰포트에 맞게 터미널을 조정한다.

- 터미널 폰트 크기/줄 수를 모바일 뷰포트에 맞게 조정
- xterm.js cols/rows 재계산 (모바일 화면 크기 기준)
- 모바일 키보드 표시 시 터미널 영역 리사이즈
- 화면 회전(가로 모드) 시 터미널 재조정 (가로 모드에서 더 넓은 cols)

### REQ-5: 모바일 Claude Code Panel

Claude Code Panel을 모바일에 맞게 재구성한다.

- 타임라인만 전체 화면 → 하단 토글로 터미널 전환
  - 타임라인 뷰와 터미널 뷰를 탭 형태로 전환 (분할 표시 아님)
- 입력창은 화면 하단 고정 (모바일 키보드 위)
- 세션 메타 바: 컴팩트 모드만 표시 (상세 확장은 별도 시트/모달)
- 세션 목록: 전체 화면 리스트로 표시

### REQ-6: 모바일 생성 기능

모바일에서도 최소한의 생성/관리 기능을 제공한다.

- 트리 탐색 메뉴에서 Pane 하위에 "+" 버튼 → 새 Surface(탭) 생성
- 현재 Surface를 닫는 버튼 (네비게이션 바 또는 트리 메뉴)
- Workspace/Pane 생성은 데스크톱에서만 (모바일에서는 불필요)

### REQ-7: 통계 페이지 모바일 대응

통계 페이지(`/stats`)도 모바일 레이아웃을 지원한다.

- 단일 컬럼 레이아웃 (데스크톱과 동일하지만 너비 100%)
- 차트: 모바일 너비에 맞게 반응형 리사이즈
- 기간 필터: 가로 스크롤 가능한 탭 버튼 또는 드롭다운으로 전환

## 비기능 요구사항

### NFR-1: 기존 데스크톱 레이아웃 영향 없음

모바일 뷰 추가가 기존 데스크톱 레이아웃과 기능에 영향을 주지 않아야 한다.

### NFR-2: 터미널 성능

모바일에서도 터미널 입출력이 데스크톱과 동일한 반응성을 유지해야 한다.

### NFR-3: 터치 친화적

모든 인터랙션이 터치로 자연스럽게 동작해야 한다. (탭, 스와이프, 스크롤, 키보드 입력)

### NFR-4: 오프라인 대응 없음

모바일 뷰는 서버 연결이 필수. PWA/오프라인 기능은 범위 외.

---

## 범위 제외 (Phase 13에서 하지 않는 것)

| 항목 | 사유 |
|---|---|
| 네이티브 앱 (iOS/Android) | 반응형 웹으로 충분 |
| PWA / 오프라인 | 터미널은 서버 연결 필수 |
| 모바일 전용 단축키 | 터치 기반이므로 불필요 |
| 모바일에서 Pane 분할 | 화면이 작아 실용적이지 않음 |
| 모바일에서 Workspace/Pane 생성 | 데스크톱에서 설정, 모바일은 조회/조작만 |

## 기술 구성

### 반응형 분기 흐름

```
페이지 로드
├── 뷰포트 너비 확인
│   ├── < 768px → 모바일 레이아웃 렌더링
│   └── >= 768px → 데스크톱 레이아웃 렌더링 (기존)
├── resize 이벤트 감지 (debounce)
│   └── 뷰포트 변경 시 레이아웃 전환
└── 터미널/WebSocket 연결은 레이아웃과 무관하게 동일
```

### 모바일 터미널 연결 흐름

```
Surface 선택 (트리 탐색 또는 스와이프)
├── 해당 Surface의 tmux 세션에 WebSocket 연결
├── xterm.js 인스턴스 마운트 (모바일 cols/rows)
├── 이전 Surface의 WebSocket은 유지 (백그라운드)
└── 스와이프로 전환 시 즉시 표시 (미리 연결)
```

## 검증 시나리오

1. **자동 분기**: 모바일 브라우저 접속 → 모바일 레이아웃 표시 (Pane 분할 없이 단일 화면)
2. **트리 탐색**: 햄버거 메뉴 → Workspace/Pane/Surface 트리 탐색 → Surface 선택 → 전체 화면 터미널
3. **터미널 동작**: 모바일에서 터미널 명령어 입력/실행 정상 동작
4. **Claude Code**: 모바일에서 타임라인 전체 화면 표시, 하단 입력창으로 메시지 전송
5. **터미널 토글**: Claude Code 모드에서 터미널 토글 → 타임라인 ↔ 터미널 전환
6. **스와이프**: 좌/우 스와이프 → 같은 Pane 내 다른 Surface로 전환
7. **모바일 키보드**: 키보드 표시 시 터미널/입력창 영역 정상 리사이즈
8. **화면 회전**: 세로 → 가로 전환 시 터미널 cols 재조정
9. **통계 페이지**: 모바일에서 `/stats` 접근 → 차트가 모바일 너비에 맞게 표시
10. **데스크톱 영향 없음**: 모바일 뷰 추가 후에도 데스크톱 레이아웃 정상 동작
