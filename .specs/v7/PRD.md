## 페이지 목록 (도출)

v7은 새로운 페이지 추가가 아닌, 기존 터미널 페이지에 단축키 시스템을 오버레이하는 작업이다.

| 페이지/모듈 | 설명 | 우선순위 |
|---|---|---|
| 단축키 시스템 (훅) | react-hotkeys-hook 기반 중앙 단축키 등록 + 키 매핑 상수 | P0 |
| xterm.js 키 이벤트 분리 | `attachCustomKeyEventHandler`로 앱 단축키를 터미널에서 분리 | P0 |
| Pane 단축키 | 분할(`⌘D`, `⌘⇧D`), 포커스 이동(`⌥⌘←/→/↑/↓`) | P0 |
| Surface 탭 단축키 | 생성(`⌘T`), 닫기(`⌘W`), 이전/다음(`⌘⇧[/]`), 번호 이동(`⌃1~9`) | P0 |

## 주요 요구사항

### 단축키 시스템 (인프라)

- `react-hotkeys-hook` (v5)의 `useHotkeys` 훅으로 각 단축키를 선언적으로 등록
- 키 조합 상수를 별도 설정 파일로 분리하여 매핑 일원화
- 각 단축키는 액션 함수를 호출 (컴포넌트 직접 참조 없음)
- `preventDefault: true`로 브라우저 기본 동작 차단 (`⌘T`, `⌘W`, `⌘1~9` 등)
- `enableOnFormTags: true`로 터미널 포커스 상태에서도 동작 보장

### xterm.js 키 입력 충돌 해결

- xterm.js의 `attachCustomKeyEventHandler`에서 앱 단축키 조합은 `return false`로 브라우저에 버블링
- 일반 키 입력은 `return true`로 터미널이 처리
- `⌘C`(복사), `⌘V`(붙여넣기)는 터미널/브라우저 기본 동작 유지 (useHotkeys에 등록하지 않음)

### Pane 분할

- `⌘D` / `Ctrl+D` — 포커스된 Pane을 수직 분할 (기존 탭 바 분할 버튼과 동일한 함수 호출, 별도 구현하지 않음)
- `⌘⇧D` / `Ctrl+Shift+D` — 포커스된 Pane을 수평 분할 (기존 탭 바 분할 버튼과 동일한 함수 호출, 별도 구현하지 않음)
- 새 Pane에 기본 탭 1개 생성, 포커스 자동 이동
- 분할 시 기존 Pane의 작업 디렉토리 유지 — 기존 분할 로직 그대로 사용

### Pane 포커스 이동

- `⌥⌘←/→/↑/↓` / `Ctrl+Alt+←/→/↑/↓` — 해당 방향의 인접 Pane으로 포커스 이동 (레이아웃 트리 기반 판정)
- 이동 시 해당 Pane의 활성 탭 터미널에 자동 포커스
- 해당 방향에 Pane이 없으면 무시
- 포커스 변경은 layout.json에 즉시 저장 (Phase 6 영속성 활용)

### Surface (탭) 조작

- `⌘T` / `Ctrl+T` — 포커스된 Pane에 새 탭 생성 + 활성화
- `⌘W` / `Ctrl+W` — 포커스된 Pane의 활성 탭 닫기 (기존 탭 바 X 버튼과 동일한 함수 호출, 코드 중복 없이)
  - 마지막 탭 닫기 → Pane 자동 제거, 인접 Pane으로 포커스 이동
  - Workspace의 최후 Pane/탭 → 기존 X 버튼 동작과 동일하게 처리
- `⌘⇧[` / `Ctrl+Shift+[` · `⌘⇧]` / `Ctrl+Shift+]` — 이전/다음 탭 전환 (끝에서 순환하지 않음)
- `⌃1~8` / `Alt+1~8` — N번째 탭으로 이동
- `⌃9` / `Alt+9` — 마지막 탭으로 이동
- 존재하지 않는 번호의 탭은 무시

### Workspace 전환

- `⌘1~8` / `Ctrl+1~8` — N번째 Workspace로 전환 (사이드바 목록 순서 기준)
- `⌘9` / `Ctrl+9` — 마지막 Workspace로 전환
- 존재하지 않는 번호는 무시
- 전환 시 해당 Workspace의 Pane 레이아웃 + 포커스 Pane 복원

### 터미널 스크롤백 지우기

- `⌘K` / `Ctrl+K` — 포커스된 터미널의 스크롤백 버퍼 초기화 (xterm.js `clear()` 호출)

## 제약 조건 / 참고 사항

- **브라우저 단축키 차단**: `⌘T`, `⌘W`, `⌘1~9` 등은 `preventDefault`로 브라우저 기본 동작을 차단함. 브라우저 자체 탭/창 조작은 불가해짐 — 이는 cmux 등 동일 유형 앱의 표준 동작
- **크로스플랫폼 키 매핑**: macOS는 `⌘`(meta), Windows/Linux는 `Ctrl`로 매핑. 키 매핑 상수에서 OS를 감지하여 분기 처리
- **Pane 포커스 방향 판정**: 레이아웃 트리(split/pane 노드) 기반으로 인접 Pane을 판정
- **성능**: 단축키 입력 → 액션 실행은 동기적으로 즉시 처리. API 호출(layout 저장)은 비동기이되, UI는 optimistic update로 즉시 반영
- **기존 마우스 조작과의 호환**: 단축키 시스템 추가가 기존 마우스 클릭/드래그 동작에 영향을 주지 않아야 함
