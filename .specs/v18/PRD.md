# v18 요구사항 정리

## 출처

- `.specs/v18/requirements/my-agent.md` — My Agent 자율 코딩 에이전트 전체 설계

## 페이지 목록 (도출)

| 페이지 | 설명 | 우선순위 | Phase |
| ------ | ---- | -------- | ----- |
| 에이전트 목록/관리 | 에이전트 생성, 설정, 삭제. 역할/담당 프로젝트 지정 | P0 | 1 |
| 에이전트 채팅 | 에이전트와 1:1 대화. 지시, 질문 응답, 보고 수신 | P0 | 1 |
| 미션/태스크 현황 | Mission → Task → Step 트리 뷰. 상태 머신 시각화 | P1 | 2 |
| 에이전트 워크스페이스 | 에이전트 전용 워크스페이스. 계획/관리 공간 | P1 | 2 |
| 메모리 뷰어 | 에이전트 메모리/학습 내용 열람 및 편집 | P2 | 4 |

## 주요 요구사항

### 에이전트 목록/관리 (Phase 1)

- 에이전트 생성: 이름, 역할(자유 텍스트), 담당 프로젝트 지정
- 에이전트 설정 파일(`~/.purplemux/agents/{agent}/config.md`)로 역할, 담당 프로젝트, 자율성 수준 저장
- 에이전트 삭제 시 관련 세션/메모리 정리
- 에이전트 상태 표시: idle / working / blocked
- 헤더에서 에이전트 목록 진입점 제공

### 에이전트 채팅 (Phase 1)

- 에이전트별 독립 채팅 UI (인턴과 메신저하는 느낌)
- **턴 기반 통신**: 한 번에 하나의 메시지, 응답 대기 후 다음
- 사용자 → 에이전트: 서버가 에이전트 idle 확인 후 tmux send-keys로 전달
- 에이전트 → 사용자: 에이전트가 `POST /api/agent/message` 호출 → 서버가 WebSocket으로 채팅 UI에 push
- 에이전트가 먼저 말 걸 수 있음 (질문, 중간보고, 완료보고, 승인 요청)
- 메시지 유형 구분: `report`, `question`, `done` 등 → 중요도 필터링 가능
- 채팅 메시지 영속화: 세션별 JSONL (append-only)
  ```
  ~/.purplemux/agents/{agent}/chat/
  ├── index.json          # 세션 목록 (메타데이터)
  └── {sessionId}.jsonl   # 메시지 한 줄씩 append
  ```
- 서버가 양방향 메시지를 모두 기록 (에이전트는 저장 신경 안 씀)

### 에이전트 실행 구조 (Phase 1)

- 에이전트 자체가 Claude Code 세션으로 동작 (Claude API가 아닌 Claude Code 사용, 비용 효율)
- 서버는 얇은 중계 레이어: 메시지 전달, 상태 관리, 세션 복구만 담당
- 에이전트가 tmux를 직접 제어하여 탭 생성 / Claude Code 실행
  - 탭 생성 → tmux API로 세션/윈도우 생성
  - Claude Code 실행/입력 → tmux send-keys
  - 상태 확인 → 기존 상태 감지 로직 재사용 (pane title, 세션 감지)
  - 결과 확인 → jsonl 파싱 (필요시 tmux 버퍼 보조)
- 완료/실패 감지: 기존 상태 감지 로직 재사용
- 사람이 purplemux를 쓰는 방식과 완전히 동일하게 동작 (에이전트 전용 분기 없음)
- CLAUDE.md를 통해 프로젝트별 시스템 프롬프트 적용
- 에이전트 세션은 도구 승인 프롬프트 없이 동작 (자동 승인 설정)
- 에이전트 전용 CLAUDE.md로 역할, 도구 사용법, 보고 규칙 정의

### 미션/태스크 현황 (Phase 2)

- 3단계 트리 구조: Mission → Task → Step
- 상태 머신: `pending → running → completed / blocked / failed`
- `blocked` 상태 시 해당 태스크만 멈추고, 다른 태스크는 계속 진행
- blocked 발생 시 사용자에게 알림 전송
- 병렬 태스크: 탭 여러 개 동시 실행
- 태스크 진행 상황 UI (트리 뷰)
- Phase 2에서 미션 연결 시 채팅 index에 missionId 필드 추가

### 자율 계획 (Phase 3)

- Mission을 받으면 에이전트가 Task 목록을 자동 생성
- 롤링 계획: 처음부터 전체를 확정하지 않고, 앞 단계 결과를 반영하여 뒷 단계를 조정
- 각 Task 시작 시 Step 목록을 구체화
- Step 결과를 파일로 저장 → 다음 Step에 컨텍스트 전달 (컨텍스트 유실 방지)
- 자율성 경계: 초반 보수적(많이 물어봄) → 메모리에 "이런 건 알아서 해도 됨" 축적하며 점진적 확대

### 메모리 시스템 (Phase 4)

- 파일 기반 영속 메모리 (`~/.purplemux/agents/` 하위)
- 계층 구조:
  - `shared/user.md` — 사용자 선호도, 작업 스타일 (반영구)
  - `shared/global-learnings.md` — 공통 교훈 (반영구)
  - `shared/projects/{project}/context.md` — 기술 스택, 컨벤션 (프로젝트 수명)
  - `shared/projects/{project}/learnings.md` — 프로젝트별 교훈 (누적)
  - `shared/projects/{project}/index.md` — 주요 파일/모듈 맵, git diff 기반 갱신
  - `{agent}/config.md` — 역할, 담당 프로젝트, 자율성 수준
  - `{agent}/memory/` — 에이전트별 경험/판단 기록 (누적)
  - `{agent}/missions/` — 미션 이력 아카이브
- 에이전트가 스스로 메모리 작성/갱신/정리
- 파일 = 메모리 단위, 느슨한 연결 (디렉토리 컨벤션만), 별도 DB 없음
- 메모리 뷰어 UI로 사용자가 열람/편집 가능

### 멀티 에이전트 (Phase 5)

- 에이전트 여러 개 생성/운영
- 역할별 분리 (백엔드 전담, 프론트 전담, 리뷰어 등)
- 프로젝트별 분리 (같은 프로젝트에 여러 에이전트 가능)
- 에이전트별 독립 워크스페이스
- 공유 메모리 + 크로스 학습:
  - 프로젝트 메모리는 모든 에이전트가 읽고 쓸 수 있음
  - 에이전트 A가 배운 범용 교훈 → shared로 승격
  - 에이전트별 memory/는 해당 에이전트만의 맥락 (다른 에이전트가 참조는 가능)

## 제약 조건 / 참고 사항

- **Claude Code 기반 실행**: Claude API 직접 호출 아닌 Claude Code 세션 사용 (비용 효율)
- **기존 인프라 최대 활용**: tmux 제어, 상태 감지, 워크스페이스/탭 구조 등 purplemux 기존 코드 경로 재사용
- **에이전트 전용 분기 최소화**: 사람과 에이전트가 같은 코드 경로를 사용
- **서버는 오케스트레이터가 아님**: 메시지 전달과 상태 영속화만 담당, 판단/계획은 에이전트가 수행
- **권한 제한 없음**: 우선은 에이전트가 모든 작업 수행 가능 (Phase 3에서 자율성 경계 도입)
- **상태 저장**: JSON 파일, 디렉토리 단위로 분리 (`~/.purplemux/agents/` 하위)
- **Phase 순서**: Phase 1→2→3 순차, Phase 4는 3 이후 아무때나, Phase 5는 4 필요
- **응답 지연 고려**: 에이전트 Claude Code 세션 응답이 느릴 수 있으므로 채팅 UI에서 로딩/타이핑 인디케이터 필요
- **세션 복구**: 에이전트 세션이 죽으면 서버가 자동 재시작

## 미확인 사항

- [ ] 에이전트 동시 실행 수 제한이 필요한지 (리소스/비용 관점)
- [ ] 에이전트가 생성한 PR/커밋에 대한 사용자 승인 워크플로우 (자동 머지 허용 범위)
- [ ] 병렬 태스크 시 동시성 제어 — 에이전트 세션에서 상태 폴링을 어떻게 처리할지
- [ ] 에이전트 간 충돌 방지 — 같은 프로젝트에 여러 에이전트가 같은 파일을 수정할 때
- [ ] claude -p 출력 파싱 전략 구체화 — CLAUDE.md에 JSON 응답 규칙 넣는 것만으로 충분한지
- [ ] 에이전트 비활성 타임아웃 — 장시간 idle 에이전트 자동 종료 정책
- [ ] 에이전트별 비용 추적 — Claude Code 사용량 모니터링/제한
- [ ] 메모리 크기 관리 — 메모리 파일이 과도하게 커졌을 때의 정리/요약 전략
- [ ] 에이전트 롤백 — 에이전트가 잘못된 작업을 했을 때 되돌리는 메커니즘
- [ ] 알림 채널 — blocked/완료 알림을 purplemux 내부 알림 외에 외부(Slack 등)로도 보낼지
