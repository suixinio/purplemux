# CLAUDE.md

This document defines the rules and conventions Claude Code must follow when working on this project.

AI-generated code should look as if a senior engineer wrote it. Avoid excessive comments, unnecessary explanations, and AI-specific patterns.

Always respond in the same language the user wrote in. Project rules like "commit messages in English" apply only to the artifacts they describe — they do not change the conversation language.

## Project Overview

- **Framework**: Next.js (Pages Router)
- **Package manager**: pnpm
- **Styling**: Tailwind CSS v4 + shadcn/ui
- **Language**: TypeScript

## Port

The default port is **8022** (a portmanteau of web + ssh). It can be overridden via the `PORT` env var in `server.ts`.

## Core Rules

### 1. Next.js Configuration

- Use the Pages Router (not the App Router)
- Do not use the `"use client"` directive
- Starting with Next.js 16, `middleware.ts` was renamed to **`proxy.ts`**. In this project, `src/proxy.ts` serves as the auth middleware

### 2. Package Management

```bash
pnpm add <package>        # install package
pnpm add -D <package>     # install dev dependency
pnpm dev / pnpm build / pnpm lint
```

**Do not use npm or yarn. Always use pnpm.**

### 3. File Naming Convention

All filenames use **lowercase** and **dashes (-)**:

```
✅ header-navigation.tsx, use-auth.ts
❌ HeaderNavigation.tsx, useAuth.ts
```

### 4. TypeScript Conventions

#### Interface / Type

```typescript
interface IUser {
  id: string;
  name: string;
} // I prefix
type TUserRole = 'admin' | 'user' | 'guest'; // T prefix, union type
```

#### Function Definition Style

Use **arrow functions** instead of the `function` keyword:

```typescript
// ✅ preferred
const handleClick = () => {
  // ...
};

const fetchData = async () => {
  // ...
};

// ❌ avoid
function handleClick() {
  // ...
}
```

#### Import Path

Use the **alias (`@/`)** instead of relative paths (`../`):

```typescript
import { Button } from '@/components/ui/button';
import { useAuth } from '@/hooks/use-auth';
```

#### Comment Policy

If a comment is necessary, you must be able to justify why. If you cannot, delete it.

```
✅ Allowed:
- JSDoc/TSDoc-style API documentation
- Standard markers like TODO, FIXME, HACK
- Explanations of "why" for complex business logic
- Hard-to-read code such as regex or magic numbers

❌ Forbidden:
- Comments that describe what the code does (the code itself should explain)
- Comments that restate variable or function names
- Meta comments like "this code was generated by AI"
- Decorative section dividers (e.g. //======)
```

### 11. React useRef Policy

Only use `useRef` when truly necessary; prefer alternatives where possible:

```
✅ useRef allowed for:
- Direct DOM access (focus, scroll, measurement, etc.)
- Managing external library instances
- Cases where comparing the previous value is essential

❌ Use other approaches instead of useRef for:
- State management → useState
- Derived values → useMemo
- Stabilizing callbacks → useCallback
- Component-to-component communication → props, context
```

### 13. Terminal Implementation

When writing terminal-related code, refer to **VSCode's integrated terminal implementation**:

- Follow VSCode's approach for terminal UI/UX, I/O handling, ANSI escape sequence parsing, etc.
- Prefer the patterns VSCode adopts for xterm.js-based rendering, buffer management, and key bindings
- When designing terminal features, use the VSCode source (`vscode/src/vs/workbench/contrib/terminal/`) as a reference

### 14. Terminal Process / Path Detection

When dealing with terminal process names, paths, or Claude session state, **always reuse existing utilities**. Do not call `pgrep`, `ps`, or `lsof` directly.

#### tmux Title Format

tmux sends the title in the format `"#{pane_current_command}|#{pane_current_path}"` (`src/config/tmux.conf`). This value is delivered in real time via xterm.js `onTitleChange`.

#### Client (browser)

| Function | File | Purpose |
| --- | --- | --- |
| `parseCurrentCommand(raw)` | `lib/tab-title.ts` | Extract process name from title |
| `isShellProcess(raw)` | `lib/tab-title.ts` | Detect whether the foreground process is a shell |
| `formatTabTitle(raw)` | `lib/tab-title.ts` | Convert to tab display name |
| `onTitleChange` | `hooks/use-terminal.ts` | Receive title change events |

#### Server (Node.js)

| Function | File | Purpose |
| --- | --- | --- |
| `getPaneCurrentCommand(session)` | `lib/tmux.ts` | Get foreground process name |
| `getSessionCwd(session)` | `lib/tmux.ts` | Get current working directory |
| `getSessionPanePid(session)` | `lib/tmux.ts` | Get the pane's shell PID |
| `checkTerminalProcess(session)` | `lib/tmux.ts` | Safe shell check (used before resume) |
| `isClaudeRunning(panePid)` | `lib/session-detection.ts` | Check whether a Claude process is running under the pane |
| `detectActiveSession(panePid)` | `lib/session-detection.ts` | Detect Claude sessions (`active`/`none`/`not-installed`) |
| `watchSessionsDir(panePid, cb)` | `lib/session-detection.ts` | Watch for session start/stop |
| `isProcessRunning(pid)` | `lib/session-detection.ts` | Check whether a PID is alive |

```
✅ Use existing functions:
- Process check → getPaneCurrentCommand() or onTitleChange + parseCurrentCommand()
- Path lookup → getSessionCwd()
- Claude detection → isClaudeRunning() / detectActiveSession() (server)
- Safe shell check → checkTerminalProcess()

❌ Do not implement directly:
- Calling ps, pgrep, lsof via child_process
- Writing new tmux commands (use existing functions in lib/tmux.ts)
- Duplicating process polling logic
```

### 17. Version Bumping

Use **`npm version`** to bump the package version. Do not edit `package.json`'s `version` field by hand.

```bash
npm version patch   # 1.0.0 → 1.0.1
npm version minor   # 1.0.0 → 1.1.0
npm version major   # 1.0.0 → 2.0.0
npm version prerelease --preid=beta  # 1.0.0 → 1.0.1-beta.0
```

This updates `package.json` and creates a git tag in one step, keeping the tag and version in sync.

### 18. Module Isolation Between Custom Server and Next.js

In **both** dev and production, the outer custom server (`server.ts`) and Next.js (pages + API routes) use **separate module graphs** inside the same Node process. Any `src/lib/*` module imported from both sides is instantiated **twice** — same source, **independent module-level state**.

- **Dev** (`pnpm dev`): `tsx watch server.ts` loads `server.ts` through Node's loader, while Next dev compiles pages and API routes with its own bundler (Turbopack/webpack). HMR can re-evaluate a Next-side module on file change, resetting its state again.
- **Production** (`pnpm start`): `tsup` bundles `server.ts` into `dist/server.js` (with `src/lib/*` inlined), and `next build` bundles API routes into `.next/standalone/.next/server/chunks/*.js` (with their own inlined copy of the same `src/lib/*`). `dist/server.js` `require()`s the standalone entry, so both live in one process — but Node's module cache keys on file path, and the two copies sit in different files.

Shared `node_modules` (e.g. `ws`, `pino`) resolve to a single instance because both bundlers leave them `external`. Anything authored under `src/` does not.

#### When to consider

Any **module-level value with identity or side effects** that must be seen by both the custom server and Next API routes:

- Singletons — managers, stores (e.g. `StatusManager`)
- Collections — `Map`/`Set` of WebSocket clients, sessions, locks
- Mutex/lock chains — `Promise` chains used to serialize file writes
- Caches — content/ETag/dedupe caches
- Side-effecting singletons — loggers with transport workers (pino), file watchers, timers
- Tokens/secrets generated once per process (e.g. CLI token)

Symptoms of getting this wrong: state written from `server.ts` is invisible to API routes, locks don't actually serialize writes, two transport workers fight over the same log file, a WebSocket broadcast reaches only half the clients.

Pure functions, types, and constants do **not** need this treatment.

#### Solution: namespace on `globalThis`

`globalThis` is the one object guaranteed to be shared across both module graphs within a single Node process. Hang shared state off it under a `__pt` prefixed key, guarded so re-evaluation doesn't overwrite it.

```typescript
// ✅ shared across server.ts, API routes, and HMR reloads
const g = globalThis as unknown as { __ptFooStore?: Map<string, IFoo> };
if (!g.__ptFooStore) g.__ptFooStore = new Map();
const store = g.__ptFooStore;

// ✅ singleton accessor
const g = globalThis as unknown as { __ptBarManager?: BarManager };
export const getBarManager = (): BarManager => {
  if (!g.__ptBarManager) g.__ptBarManager = new BarManager();
  return g.__ptBarManager;
};

// ❌ module-level — reset on HMR, duplicated across module graphs
const store = new Map<string, IFoo>();
```

#### Conventions

- **Key prefix**: `__pt` + PascalCase (e.g. `__ptStatusManager`, `__ptConfigLock`). Older code uses `__purplemux_` — keep existing keys as-is, but use `__pt` for new entries.
- **Guard every write**: `if (!g.__ptX) g.__ptX = ...` — never unconditionally assign, or HMR will clobber live state.
- **Type the cast once per file**: `const g = globalThis as unknown as { __ptX?: T }` at the top, then use `g.__ptX` inline.
- Existing examples: `src/lib/status-manager.ts`, `src/lib/config-store.ts`, `src/lib/sync-server.ts`, `src/lib/terminal-server.ts`, `src/lib/logger.ts`, `src/lib/cli-token.ts`.
